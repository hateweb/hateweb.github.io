the neverending problems of void gnu/linux

happy new year to everyone who's reading this. i have decided that my artix system has become way too stable. "new year, new distro!" i thought. void linux (glibc) seemed promising, since they're the people behind runit, a great init system. and believe me, the pid1 is one of the very few good things about this distro. since january 1st, i've been fighting all these problems, and some of them are just laughable for a modern linux system.

buildup

the live iso uses dash as the default shell, (which is not intended to be used as an interactive shell, as it doesn't have basic features like autocompletion), even though bash is preinstalled.

the so-called void-installer has very little configuration available, and doesn't even let you configure encryption or the efi mountpoint. the installer shouldn't exist in the first place in such a distribution.

there is a global system config at /etc/rc.conf, where you can set the timezone, hostname, etc., but the handbook explicitly tells to NOT use this config, and to configure these things the classic way. why does /etc/rc.conf exist?

culmination

there is an efibootstub hook that is supposed to eliminate the need for an external bootloader. it should automatically add an efi boot entry, but it keeps breaking all the time, so i just gave up completely on using that. the kernel and initramfs filenames are inconsistent (vmlinuz-x.x instead of vmlinuz), which is why the efibootstub hook exists in the first place. maybe you guys should just make the kernel have a persistent filename, just like every normal distribution does? not to mention the harm of writing to the motherboard's NVRAM every minor version bump. a proper operating system is supposed to create a boot entry once, and never change it.

the base metapackage has a whole bunch of unnecessary packages, while the base-minimal metapackage is made for docker containers, and is strongly advised not to use on real hardware. the void maintainers recommend installing base, and then forcefully removing all the mess, and putting the unwanted packages into the IgnorePkg list. just like microsoft windows does it. we've come full circle, forcefully debloating linux DIY distributions now.

the official software repos use let's encrypt for their ssl certificates, and i've already witnessed them expire, which is just an absolute joke for such a distro.

the biggest problem: software repositories

the repos barely have any packages and contain only the bare minimal for a working system, and there's no such thing as a centralized user repository. there is a separate repository, which requires building everything from source. there's no need for an explanation on how impractical that is. also gentoo exists.

the repository maintainers remove or refuse to add packages because of personal hostility towards the developers of that software, making up insane reasons as to why they are not planning to add it. (hyprland) the linux package contains a 1.5 year old version of the kernel. why? "stability". funny enough, there is the linux-lts package for it, the whole goal of which is being stable. but if you want at least a relatively new version of the kernel, you'll need the linux-baseline package, the description of which says that this version is "for experts only". in a DIY distribution.

conclusion

void claims to be a "stable-rolling" release distribution (whatever that means), yet it's the least stable linux distribution i've used in quite a while. this false sense of "stability" makes using the distro an incredibly unpleasant experience.
